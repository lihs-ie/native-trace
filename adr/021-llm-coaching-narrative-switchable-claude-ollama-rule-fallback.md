# LLM coaching narrative — switchable claude -p / Ollama provider with rule-based fallback

ADR-021: 録音固有 LLM コーチング文の生成 — provider 切替（claude -p / Ollama）と決定論フォールバック

# Status

Proposed

# Context

ADR-004 は worker が structured diff（phenomenon + gop + messageJa=null）を返し、frontend の ImprovementMessageGenerator が messageJa と 3 層フィードバック（whatJa/whyJa/howJa）を埋める設計を確定した。ADR-004 Context L18 は『improvement messages are generated in the frontend with a switchable RuleBased/LLM strategy』と書きつつ、REQ-106 により LLM 経路を DEFERRED として棚上げしていた。本 ADR はその defer を解除する（ユーザー確定 2）。

現状の実装はルールベース 1 本。applications/frontend/src/acl/improvement-message/rule-based/create-rule-based-improvement-message-generator.ts（実測 219 行、generate は generateFeedbackLayersFromInput(input).whatJa を返し L207-219 で factory + plain object を export）が phenomenon × catalogId × wordPair × 位置 でテンプレ文を組み立てる。決定論で安定だが、howJa は catalog.articulation.stepsJa[0..2] を機械連結するだけで、学習者の実際の録音（どの代替音を出したか=detectedTopCandidate、GOP がどれだけ低いか）に narrative を仕立てられない。この既存ルールベース生成器（本番実コード）が本 ADR の実フォールバックの正本であり、別途 backbone ADR を前提にしない。

ユーザー確定（grill ロック済み）: 確定 2=HOW 生成は『決定論ルール/カタログ backbone + 学習者の実録音に合わせた LLM narrative』で ADR-004 の LLM 戦略を un-defer。確定 4=LLM provider は Claude と local Ollama で switchable。Claude のときは課金される Anthropic API ではなくユーザー subscription 上の claude -p（headless/print mode）で起動。どちらも無いときは決定論ルールベースに fallback。

provider 機構は実機検証済み（4 番目調査、本セッション 2026-06-18 実測）。claude -p --output-format json --no-session-persistence は exit 0 で { type:result, result:<text>, total_cost_usd:... } を返し、認証は keychain（subscription）から自動取得され ANTHROPIC_API_KEY を必要としない。実測 wall time は cold ~8.8s / warm API ~4.1s。total_cost_usd（例 0.14）は keychain/subscription 認証下でも非ゼロで返るが、これは表示用の notional 値であり、ANTHROPIC_API_KEY を渡さない subscription 経路では実際の従量課金は発生しない。よってこのフィールドは確定 4『subscription であって metered API でない』と矛盾しない。Ollama は POST http://localhost:11434/api/generate（stream:false）で { response } を返すが本環境では daemon 未起動（curl exit 7）= 純粋にオプショナルなローカルサービス。

ポート現状（オンディスク確認済み）と grounding 入力の不足: usecase/port/improvement-message-generator.ts（L7-23）の ImprovementMessageGeneratorInput は phenomenon / expected{text,ipa} / detected{text,ipa} / wordPositionLabel? / catalogId? / wordPair? / expectedPronunciation? / insertedVowel? / insertionPositionMs? までしか持たず、gop / detectedTopCandidate / nBest / functionalLoad を持たない。これらは run-assessment-job/index.ts の finding 構築点では findingDraft.gop（L620 が参照）・findingDraft.detectedTopCandidate（L639-640）・findingDraft.nBest（L641）・findingDraft.functionalLoad（L643）として既にスコープ内に存在するが、generator 呼び出し（L581-594 の generate / L597-614 の generateFeedbackLayers）には渡されていない。よって LLM が録音固有事実で grounding/キャッシュするにはポート入力型を拡張し呼び出し点で findingDraft から流す必要がある（D2）。第 4 調査の codeSeams が『ポートは detectedTopCandidate を既に持つ』と記したのは誤りで、本 ADR はその前提を採らずポートを拡張する。

制約: agent-policy（AGENTS.md）=本番に mock/stub を入れない。ルールベース生成器は実コードでこれを fallback にすることでポリシー充足。REQ-106（docs/01-requirements/pronunciation-feedback-requirements.md L97-104）=LLM 生成文がカタログ整合を逸脱しないようプロンプトにカタログ項目を埋める / LLM 無しでも REQ-104 が完全機能。ast-grep environment-access-only-in-config（リポジトリルート .ast-grep/rules/environment-access-only-in-config.yml。sgconfig.yml の ruleDirs:[.ast-grep/rules] 配下で frontend 配下ではない）=process.env は infrastructure/config/** 以外で参照禁止。よって provider 設定・claude path は registry（config 経由）で読み ACL adaptor には deps として渡す。ast-grep no-class-declaration（同ルートディレクトリ）=クラス構文禁止、factory + plain object を踏襲。ポート ImprovementMessageGenerator（L31-34）は完全同期（generate は string、generateFeedbackLayers は FeedbackLayersOutput）。LLM 呼び出しは subprocess/network=非同期でポート拡張が必要。

スコープ境界: 本 ADR は PPC コンテキストのフィードバック narrative 生成（ImprovementMessageGenerator）にのみ LLM を導入する。ADR-010 が禁じる『diagnostic→weakness→focus の derivation に LLM を使わない』（ADR-010 L51 / L107-108）とは対象が重ならない（derivation は Training コンテキスト、本 ADR は PPC narrative）。本 ADR は derivation 経路に一切触れない。

ACOUSTIC grounding の扱い: 学習者録音の formant 偏差方向 / 調音方向アドバイスは、まだ未着地の acoustic-phonetic 計測 ADR（本 ADR 着地後に別 number で起草予定）が供給する構造化文字列に依存する。本 ADR はそれを任意の入力として扱い、供給されないときは ACOUSTIC キーを省くだけで完全に動く。したがって本 ADR は未存在 ADR への hard dependency を持たない。

# Decision

**D1 — provider 切替 LLM adaptor を新設し、既存ルールベース生成器（本番実コード、別 ADR 不要）を実フォールバックとして包む。** 新 ACL applications/frontend/src/acl/improvement-message/llm/create-llm-improvement-message-generator.ts を factory + plain object で実装。署名は createLlmImprovementMessageGenerator(deps:{ provider:"claude-code"|"ollama"; invoker:LlmNarrativeInvoker; cache:LlmNarrativeCache; fallback:ImprovementMessageGenerator(=createRuleBasedImprovementMessageGenerator()); promptVersion:string; providerModel:string }):ImprovementMessageGenerator。fallback は本番実コードで stub ではない。LLM 呼び出しが error/timeout/空 result/grounding 検証失敗（D4）のいずれでも fallback.generateFeedbackLayers(input) の結果をそのまま返す。

**D2 — ポート入力に grounding/cache 用フィールドを追加し、非同期メソッドを追加し、batch 並列で実行する。** usecase/port/improvement-message-generator.ts の ImprovementMessageGeneratorInput 型に任意フィールドを追加（既存 9 フィールドは不変、任意なので後方互換）: gop?:number|null; detectedTopCandidate?:string|null; nBest?:ReadonlyArray<{phoneme:string;confidence:number}>|null; functionalLoad?:string|null。これらは LLM grounding（D4 の FINDING）とキャッシュ署名（D5）に必須。run-assessment-job/index.ts の 2 つの呼び出し点（L581-594 の generate 用 input、L597-614 の generateFeedbackLayers 用 input）に、既にスコープ内の findingDraft.gop（L620 が参照）/ findingDraft.detectedTopCandidate（L639-640）/ findingDraft.nBest（L641）/ findingDraft.functionalLoad（L643）を ?? null で追加で渡す。ルールベース生成器はこれら任意フィールドを無視するため挙動不変。同型 ImprovementMessageGenerator（L31-34）に optional 非同期メソッドを追加: generateFeedbackLayersAsync?:(input:ImprovementMessageGeneratorInput)=>Promise<FeedbackLayersOutput>。ルールベース生成器は未実装（undefined=既存挙動不変）、LLM 生成器は実装。run-assessment-job の finding ループは findingsWithId 構築前に generateFeedbackLayersAsync があれば全 finding input を作り、子プロセス同時起動の暴発を避けるため並列度上限（既定 3、config llmNarrativeMaxConcurrency）でチャンク化して Promise.all で並列生成し Map<findingIndex,FeedbackLayersOutput> に貯める。ループ内 feedbackLayers 解決は findingDraft.feedbackLayers ?? precomputed.get(index) ?? generateFeedbackLayers(input)、messageJa は解決済み feedbackLayers.whatJa を採用（ルールベースの generate が generateFeedbackLayersFromInput(input).whatJa を返すのと同じ規約 L207-210）。generateFeedbackLayersAsync 未定義（ルールベース選択）時は現状同期ループ（L581-614）を維持。

**D3 — claude -p subprocess 機構（subscription、metered API 不使用）。** claude-code-narrative-invoker.ts を新設。Node child_process.spawn（exec 不可）で spawn(claudeExecutablePath,["-p","--output-format","json","--no-session-persistence","--system-prompt",systemPromptText,"--model",providerModel(既定 sonnet),userPromptText],{ env:processEnvWithoutAnthropicApiKey }) を起動。--bare は使わない（--bare は keychain/OAuth を読まず ANTHROPIC_API_KEY 必須になり subscription 経路が壊れる）。default mode で keychain（subscription）認証を継承。env に ANTHROPIC_API_KEY を渡さない（metered 経路の混入防止）。claudeExecutablePath は config から registry 経由で deps に渡す（ACL で process.env 直読み禁止）。stdout を Buffer 収集し JSON parse、result フィールドを取り出す。返却 JSON の total_cost_usd は notional 表示値で課金実体ではない（Context 参照）ため参照しない。result を JSON.parse して {whatJa,whyJa,howJa} を得る（system prompt が JSON 限定出力を強制 D4）。AbortController で llmNarrativeTimeoutMilliseconds（既定 30000ms。**ADR-023 で 60000ms に改訂** — 実測 grounded narrative ~40s/call）の timeout、超過時は子プロセスへ SIGTERM。timeout/非 0 exit/parse 失敗は reject → D1 で fallback。Docker 制約: keychain は container に存在しない。provider="claude-code" は local dev 限定。registry がこれを判定し Docker 環境（NEXT_RUNTIME 等で判定）で claude が PATH に無いとき provider を rule-based に強制 downgrade してログに記録（narrative が空にならないことを保証）。

**D4 — grounding contract（LLM は transform のみ、phonetics を invent しない）。** grounding-prompt.ts を新設。system prompt（固定文、英語）: "You are a pronunciation coach for Japanese (L1) speakers learning English. You receive a structured FINDING object and a CATALOG object. Output ONLY a JSON object with exactly three string fields whatJa, whyJa, howJa, all in Japanese. You MUST NOT introduce any phonetic claim, IPA symbol, articulatory direction, formant value, or word form not present in the FINDING or CATALOG objects. Use only the IPA symbols and words given. If a field cannot be grounded, copy the corresponding FALLBACK text verbatim. Do not add markdown, preamble, or commentary."。user prompt は次の構造化事実だけを JSON 直列化して渡す（学習者の実録音に固有）: FINDING={phenomenon,expected{text,ipa},detected{text,ipa},wordPositionLabel,gop,detectedTopCandidate,nBest[{phoneme,confidence}],insertedVowel,insertionPositionMs,wordPair,expectedPronunciation,functionalLoad}（すべて D2 で拡張した入力フィールドから供給。port 拡張前は gop/detectedTopCandidate/nBest/functionalLoad が無く grounding 不能なので D2 は本 ADR の前提条件）。ACOUSTIC（acoustic-phonetic 計測 ADR が供給する場合のみ。供給されないときはキー自体を省く）=formant 偏差方向（F1/F2/F3 の高低）と調音方向アドバイス文字列。LLM はこれらを言い換える（transform）だけで新しい formant 値や調音方向を作らない。CATALOG={l1MechanismJa,articulation.stepsJa,articulation.mannerJa,confusionSet,functionalLoad,intelligibilityImpact}（domain/error-catalog の ErrorCatalogEntry から取得）。FALLBACK=ルールベース生成器が同じ input に対して返す {whatJa,whyJa,howJa}（grounding 不能時の copy 元 + 出力検証の基準）。出力検証（採用前に必須）: 返り値が (a) 厳密に 3 キー whatJa/whyJa/howJa を持つ object、(b) 各値が空でない string、(c) 各値長が 4..400 文字、(d) FINDING/CATALOG/ACOUSTIC に出現しない IPA 風トークン（/.../ で囲まれたスラッシュ記法）を含まない、を満たすこと。1 つでも違反したら fallback を返しキャッシュしない。

**D5 — finding 署名キャッシュ（決定論 + コスト削減）。** narrative は (prompt version + provider/model + finding 構造) が同じなら決定論的に同一でよいので SQLite にキャッシュ。新 port usecase/port/llm-narrative-cache.ts: LlmNarrativeCache={ findBySignature(signature):ResultAsync<FeedbackLayersOutput|null,DomainError>; store(signature,layers,metadata):ResultAsync<void,DomainError> }。新 infra infrastructure/drizzle/repositories/llm-narrative-cache-repository.ts: createDrizzleLlmNarrativeCacheRepository。署名=sha256 of phenomenon|expected.ipa|detected.ipa|catalogId|wordPositionLabel|detectedTopCandidate|insertedVowel|promptVersion|providerModel（detectedTopCandidate は D2 でポートに追加されたフィールドから取得。拡張なしでは署名に含められない）。adaptor フロー: 署名計算 → findBySignature → hit なら即返す（LLM 呼ばない）→ miss なら invoker 実行 → D4 検証通過のみ store → 返す。練習セッションで同一 finding が再出する dominant case でゼロコスト。

**D6 — registry 配線と config（switchable、既定 rule-based）。** infrastructure/config/index.ts の configSchema に追加: llmCoachingProvider:z.enum(["claude-code","ollama","rule-based"]).default("rule-based")(env LLM_COACHING_PROVIDER); ollamaEndpoint:z.string().url().default("http://localhost:11434")(env OLLAMA_ENDPOINT); ollamaModel:z.string().min(1).default("llama3.1:8b")(env OLLAMA_MODEL); claudeCodeExecutablePath:z.string().min(1).default("claude")(env CLAUDE_CODE_PATH); claudeCodeModel:z.string().min(1).default("sonnet")(env CLAUDE_CODE_MODEL); llmNarrativeTimeoutMilliseconds:z.coerce.number().int().positive().default(30000)(env LLM_NARRATIVE_TIMEOUT_MS); llmNarrativePromptVersion:z.string().min(1).default("v1")(env LLM_NARRATIVE_PROMPT_VERSION); llmNarrativeMaxConcurrency:z.coerce.number().int().positive().default(3)(env LLM_NARRATIVE_MAX_CONCURRENCY)。registry.ts L357（const improvementMessageGenerator = createRuleBasedImprovementMessageGenerator();）を分岐に置換: provider==="rule-based"（既定）なら現状どおり（挙動不変）、"claude-code"/"ollama" なら createDrizzleLlmNarrativeCacheRepository(database) を構築し provider 別 invoker を組み createLlmImprovementMessageGenerator({provider,invoker,cache,fallback:createRuleBasedImprovementMessageGenerator(),promptVersion,providerModel}) を improvementMessageGenerator にする。L486 の createRunAssessmentJob({...,improvementMessageGenerator}) は無改修。Docker かつ claude PATH 不在のとき provider を rule-based に downgrade（D3）。

**D7 — Ollama invoker。** ollama-narrative-invoker.ts を新設。fetch(ollamaEndpoint + "/api/generate",{method:"POST",body:JSON.stringify({model:ollamaModel,system:systemPromptText,prompt:userPromptText,stream:false}),signal:AbortSignal.timeout(timeoutMs)}) で response.json().response を取り出し D4 と同じ検証・parse をかける。daemon 未起動（connection refused）は reject → fallback。

**D8 — UI 挙動。** narrative は既存 feedbackLayers（EngineFindingDto.feedbackLayers:FeedbackLayersDto）に乗るため DetailPanelV2 / ArticulationCard の表示は無改修で LLM 文に切り替わる。provider が rule-based のときは現状文がそのまま出る。UI には provider バッジ等を出さない（first slice では narrative の出所を学習者に見せない）。messageJa も同じ feedbackLayers の whatJa を採用。

**D9 — ADR-004 を改訂し LLM 戦略を active かつ provider-switchable と記録する。** ADR-004 Context L18 の『switchable RuleBased/LLM strategy（the worker must not embed an OpenAI client）』に『LLM 経路は本 ADR-021 により un-defer（REQ-106 の defer 解除）。LLM narrative 層は provider 切替（claude-code subscription / ollama / rule-based fallback）で active』を追記し Changes/Related に ADR-021 を追記。worker 契約（structured diff / messageJa=null）は不変。frontend が messageJa/feedbackLayers を埋める責務も不変で、その埋め方に LLM 経路が加わるだけ。

# Alternatives considered

- **metered Anthropic API（ANTHROPIC_API_KEY）で Claude を呼ぶ** — Pros: 並列化が容易、Docker でも動く。Cons: 課金される。確定 4 違反。不採用理由: ユーザー確定 4 が明示的に禁止。subscription 上の claude -p を使う。API key を env に置くと keychain 認証ではなく従量課金経路に切り替わる（claude --help: --bare 時は『Anthropic auth is strictly ANTHROPIC_API_KEY or apiKeyHelper、OAuth/keychain は読まれない』）。よって --bare は使わず default mode で keychain を使い env に ANTHROPIC_API_KEY を渡さない。
- **ポートを同期のまま spawnSync で同期 subprocess 実行** — Pros: ポート無改修。Cons: イベントループ長時間ブロック、並列不能。不採用理由: claude -p は cold ~8.8s。同期ブロックすると runAssessmentJob の job runner tick（poll 2000ms）と Node イベントループを 1 finding ごとに数秒間ブロックし複数 job 並列を不能にする。非同期 + バッチ並列が必須。
- **Option D（deferred async narrative）: job 成功後に別経路で narrative 生成し次回 read で差し替え** — Pros: job latency 不変。Cons: 二重 write path、UI 二状態、単一パスモデルを壊す。不採用理由: second write path と UI の二状態管理が増える。現状 feedbackLayers は単一パスで一度だけ set。cache hit 後はほぼ即時、cache miss でも並列度上限内で claude cold 1 回ぶん相当（best-case 楽観値、実際は spawn 起動 ~200-500ms と低スペック CPU 直列化で延びうる）に収まり lease 300s 内。複雑さに見合わない。first slice では不採用。
- **worker（Haskell）側で LLM を呼ぶ** — Pros: audio に近い。Cons: ADR-004 の責務境界違反、Haskell に subprocess/HTTP narrative 依存が混入。不採用理由: ADR-004 が『worker は OpenAI/LLM client を embed しない、messageJa は frontend が埋める』を確定済み。worker は scoring/status を所有し narrative を author しない。frontend ACL が正しい層。
- **LLM に自由生成させ catalog grounding を課さない** — Pros: narrative の自由度が高い。Cons: REQ-106 違反、誤った音声指導を学習者に教える。不採用理由: REQ-106 が catalog 整合を明示要求。小モデル（Ollama 7B）は IPA 記号や音声規則を hallucinate する（4 番目調査の risk）。本 ADR は『LLM は transform のみ、phonetics を invent しない』を grounding contract（D4）で強制する。
- **rule-based backbone を別 ADR（ADR-020 相当）として先に起草しそれに dependsOn する** — Pros: backbone 設計判断を別文書に分離できる。Cons: 未起草 ADR への依存を作り本 ADR のマージ可能性を不必要にブロックする。不採用理由: ルールベース生成器は既に 219 行の本番実コードとして存在しそれ自体が fallback の正本。fallback が『実コードである』ことは別 ADR を要さず満たされる。存在しない ADR への dependsOn ポインタを ship しないため本 ADR は self-contained とし backbone を inline 参照する。

# Consequences

## Positive

- LLM 層は ADR-004 の worker/port 責務境界を変えず、ポート入力の任意フィールド追加 + 非同期メソッド追加 + registry の 1 分岐 + ACL 追加だけで載る。既定 rule-based なので opt-in するまで既存挙動は完全不変。
- fallback が既存ルールベース実生成器（本番実コード、別 ADR を要さない）なので agent-policy（no stub）を満たし REQ-106『LLM 無しでも REQ-104 が完全機能』を構造的に保証する。
- claude -p は subscription keychain 認証で metered 課金されず（確定 4 準拠。total_cost_usd は表示値で課金実体でない）、Ollama は完全オフライン。どちらも error/timeout/grounding 違反で決定論 fallback に落ちるので壊れない。
- 署名キャッシュ（detectedTopCandidate を含む、ポート拡張後に算出可能）で同一 finding 再出時はゼロコスト・決定論。練習セッションの dominant case で LLM を呼ばない。
- narrative が既存 feedbackLayers に乗るため UI 無改修で LLM 文に切替わる。

## Negative

- claude-code provider は keychain 依存のため local dev 限定（Docker では rule-based に downgrade）。本番デプロイ形態次第で LLM narrative が出ない。
- ポート入力に 4 フィールド + 非同期メソッドを追加するため run-assessment-job ループの feedbackLayers 解決ロジックが二経路（async batch / 同期）に分岐し複雑度が上がる。
- 小モデル（Ollama 7B）の hallucination は grounding prompt + 出力検証で抑えるが完全には防げない。検証で弾いた分は fallback に落ちるので無害だが LLM の付加価値が出ない finding が残る。
- narrative の教育的有効性そのもの（特に retest ループとの相乗）はエビデンス薄。6 番目調査 E-15 のとおり『practice→retest→改善表示ループ』はピンポイント RCT が無い設計仮定。

# Compliance

- port 拡張 contract test: ImprovementMessageGeneratorInput が gop/detectedTopCandidate/nBest/functionalLoad を任意で受け、run-assessment-job が L581-614 の両 input にこれらを findingDraft から渡していること（型・値の到達）を assert。
- contract test: provider=rule-based のとき improvementMessageGenerator.generateFeedbackLayersAsync が undefined であり run-assessment-job が現状同期パスを通ること（既存 snapshot 不変）を assert。
- grounding 検証 unit test: D4 の出力検証（3 キー厳密 / 空でない / 4..400 文字 / FINDING・CATALOG・ACOUSTIC に無い /.../ IPA トークンを含まない）を満たさない LLM 返り値で fallback が返ること、検証通過時のみ cache store されることを assert。
- claude-code invoker test: spawn 引数が ['-p','--output-format','json','--no-session-persistence','--system-prompt',...,'--model',...,user] を含み '--bare' を含まず env に ANTHROPIC_API_KEY を渡さないことを assert（metered API 不使用の機械検査）。
- fallback 等価 test: invoker が reject/timeout/空 result を返したとき generateFeedbackLayersAsync の結果が fallback.generateFeedbackLayers(input) と一致することを assert。
- ast-grep: acl/improvement-message/llm/** が process.env を直読みしないこと（config は deps 経由）をリポジトリルート .ast-grep/rules/ の補助ルールで fitness 検査。LLM adaptor 内に mock/stub/fake が無いことを no-prod-doubles で担保。
- cache 決定論 test: 同一署名で 2 回呼んで 1 回目 store・2 回目 hit（invoker 非呼出）になること、promptVersion/providerModel/detectedTopCandidate が違えば別エントリになることを assert。
- drizzle migration 検査: schema.ts に llm_narrative_cache 追加後 pnpm db:generate で migration が生成され live DB に no such table が出ないこと（メモリ『drizzle migration は再生成必須』）。
- runtime 検証: provider=claude-code をローカルで有効化し実 finding を含む録音を assess して feedbackLayers が rule-based テンプレと異なる学習者固有 narrative（detectedTopCandidate を反映）になること、fallback（provider unset / daemon down / Docker downgrade）で従来文が出ること、生成文に input 外の IPA が混入しないことを live で観測 assert。

# Notes

- Risks:
  - claude -p の subscription 認証は keychain 継承に依存する。Node サーバプロセスが親と同一ユーザコンテキストでないと認証が取れない。Docker では keychain 不在 = claude-code provider 不可。registry の downgrade ロジックが正しく rule-based に落ちないと narrative が空になる恐れがある（D3 で明示・要 runtime 検証）。
  - ポート入力に grounding フィールドを追加し呼び出し点で findingDraft から流す改修が漏れると LLM は detectedTopCandidate/gop を grounding できず差別化価値が出ない（D2 が本 ADR の前提条件であることを compliance の port 拡張 test で担保）。
  - claude -p は --bare なしだと CLAUDE.md / hooks / plugin をロードするため cold 起動に ~200-500ms の追加 overhead が乗る（4 番目調査）。subprocess 起動コストは finding 並列数ぶん積算するため並列度上限（既定 3）で抑える。『5 finding 並列 ~9s』は完全並列が立つ best-case の楽観値であり低スペック CPU では直列化して延びうる。**【ADR-023 で訂正】**: 2026-06-19 の runtime 検証で grounded narrative の実測 wall time は **~40s/call** と判明（probe 値 cold ~8.8s/warm ~4.1s は本番 latency を表さない）。30000ms timeout 既定では大半が timeout→fallback（18 finding 中 LLM 1 件）。ADR-023 で timeout 既定を 60000ms に改訂し、LLM 対象 finding を `llmNarrativeMaxFindings`（既定 8、severity/functionalLoad 上位）に上限化して worst-case バッチを 300s lease 内（ceil(8/3)×60s=180s）に有界化、fallback を reason 付き構造化ログで可観測化する。
  - Ollama 7B / 小モデルの日本語音声説明品質は Claude 未満。grounding 検証で弾いた分は fallback に落ちるが LLM の付加価値が出ない finding が増える可能性（4 番目調査の risk）。
  - prompt template を変えると署名に promptVersion を含めていても旧 cache は別キーになり再生成が走る。promptVersion の bump 忘れで stale narrative が残る運用リスク。
  - ACOUSTIC grounding 入力（formant 偏差方向 / 調音方向）は acoustic-phonetic 計測 ADR（未起草）が着地するまで供給されない。本 ADR はその場合 ACOUSTIC キーを省くだけで動くが『学習者の実録音に合わせた narrative』の差別化価値はその ADR 着地まで detectedTopCandidate/gop ベースに限定される（確定 3 の深い診断は別 ADR 依存。本 ADR は hard dependency を持たない）。
  - narrative 生成の教育的有効性は SLA エビデンス薄（6 番目調査 E-15: retest ループは設計仮定）。user-facing copy で『改善が加速する』等の因果を主張しないこと。
- First-slice relevance: first slice（closed-loop minimal: A/B 部分再生 + その場 re-record → GOP delta → 改善表示）にとって本 ADR は直接の必須ではない。first slice は『部分再生 / 再録音 / GOP delta 表示』の配線が主眼で narrative の質は問わない。本 ADR の LLM 層は既定 rule-based のままでも first slice は完全に成立する（feedbackLayers はルールベースで埋まる）。よって本 ADR は first slice と並行・独立に着地でき first slice をブロックしない。ただし re-record 後の改善表示で『なぜ良くなった/まだ足りないか』を学習者の実録音（GOP delta / detectedTopCandidate）に即して語る段階では本 ADR の LLM narrative が価値を出すため first slice の次スライス（verify ループの narrative 強化）で opt-in するのが自然な順序。実装優先度は first slice 配線 → 本 ADR の port 拡張 + LLM 層、の後段に置く。
- Amends: 本 ADR-021 により ADR-004 の LLM フィードバック戦略の defer を解除。ADR-004 Context L18 の『switchable RuleBased/LLM strategy』に『ADR-021 で active、provider 切替（claude-code subscription / ollama / rule-based fallback）』を追記。worker の structured-diff / messageJa=null 契約と『frontend が messageJa/feedbackLayers を埋める』責務は不変。Related に ADR-021 を追記。

# 発音フィードバック機能 要件定義書

本書は [要件定義書 v1.0.0](./requirements-specification.md)（以下「基本要件」）を拡張し、発音フィードバックの具体性・指導性・訓練ループを定義する。根拠は [発音フィードバック改善 調査レポート](../06-research/pronunciation-feedback-research.md)（以下「調査」）の C-xx（現状課題）/ E-xx（研究知見）/ T-xx（技術）/ B-xx（商用ベースライン）にトレースする。

要件 ID は基本要件（REQ-001〜021）と衝突しない 100 番台を使用する。

## 1. 目的

現状のフィードバックは「どの音が外れたか（IPA 対比）+ 固定テンプレート文 4 種」に留まり（調査 C-1〜C-8）、ビジョン「どこが・なぜ・どのようにネイティブらしさから外れているかを把握し、反復改善でネイティブ発音を獲得する」を達成できない。本書は次の 3 つを実現する要件を定義する。

1. **フィードバックの具体化** — 「何が・なぜ・どう直すか」を説明する明示的訂正フィードバック（E-5: 説明付きフィードバックは正誤判定のみの約 1.7 倍の効果量 g=0.86 vs 0.50）
2. **採点の妥当化** — 誤りの重大度（functional load）と習熟度に基づく優先度付け（E-3, E-9, E-10）
3. **訓練ループの構築** — 評価で終わらず、診断 → 訓練 → 再評価を回す（E-1, 調査 §3.3, B-4）

## 2. ゴール設計: 二段階モデル

プロダクトビジョン「ネイティブ発音の獲得」を北極星として維持しつつ、研究コンセンサス（E-3: intelligibility 原則）と整合させるため、学習ゴールを二段階で設計する。

| 段階 | ゴール | 評価軸 | 対象となる誤り |
|---|---|---|---|
| **Stage I: 明瞭性（Intelligibility）** | 伝達を損なう誤りの解消 | error gravity 重み付けスコア、CEFR 音韻統制 3 下位尺度（全体 / 分節 / 韻律） | 高 functional load の分節誤り（/r/-/l/, 母音対立）、母音挿入、語強勢、リズム・弱形 |
| **Stage II: ネイティブ性（Nativeness）** | General American への近似 | ネイティブ近接度スコア（現行の厳しめ判定を継承） | 母音の質の精緻化、低 FL 分節（/θ/ 等）、connected speech 全現象、イントネーションの自然さ |

- ユーザーの現在段階は診断結果（REQ-121）から自動判定し、フィードバックの優先度提示（REQ-112, REQ-113）に反映する
- Stage II の項目も Stage I の段階から **検出・表示はする**が、優先度ラベルを下げ、理由（FL が低い等）を明示する（E-9: /θ/ は体感優先度と実効優先度が最も乖離する項目）

## 3. 基本要件からのスコープ変更

基本要件 §2.2「対象外」のうち、以下 3 項目は研究上フィードバック効果の中核であり（調査 C-7, E-5, §3.3）、対象外指定を撤回して本書で要件化する。

| 基本要件での対象外項目 | 撤回理由 | 本書の要件 |
|---|---|---|
| 模範音声生成および模範音声再生 | お手本との対比はフィードバックの基礎部品。OSS TTS（Kokoro-82M, Apache-2.0）で品質・ライセンスとも実現可能（T-5） | REQ-124 |
| 練習ドリル生成 | HVPT・ミニマルペア訓練は本調査で最もエビデンスが厚い手法（知覚 g≈0.67–0.92、産出転移・長期保持実証） | REQ-122, REQ-123 |
| 口の形、舌位置、息の出し方の詳細指導 | 調音指示は明示的フィードバックの中身そのもの。商用最高水準（ELSA）の差別化点（B-2, E-5） | REQ-104, REQ-105 |

その他の基本要件（教材・セクション・録音・ジョブ・履歴・比較モード等）は変更しない。

## 4. 機能要件

### 4.1 カテゴリ A: フィードバック詳細化

#### REQ-101: 日本語話者誤りカタログ <a id="req-101"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | E-8, E-9, B-3, C-2 |
| **説明** | 日本語話者の英語発音誤りを体系化したカタログをドメインモデルとして定義し、フィードバック生成・優先度付け・訓練レコメンドの共通基盤とする |
| **受入基準** | 各カタログ項目が { 種別（segmental / syllabic / prosodic）、対象音素・対立、confusion set（日本語音韻体系から導出した置換先候補）、L1 干渉メカニズムの説明文、functional load ランク（Brown 1988 準拠）、intelligibility 影響度、推奨訓練種別（知覚 / 調音 / 韻律）、エビデンス強度 } を持つこと。調査 §3.4 の誤り体系（子音 7 項目・母音 5 項目・母音挿入・韻律 4 項目以上）を初期データとして収録すること。カタログはコードから分離したデータ（編集・拡張可能）であること |

#### REQ-102: phenomenon 分類の完全実装 <a id="req-102"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | C-1, C-4, ADR-004 |
| **説明** | ADR-004 で定義済みの phenomenon enum（substitution / omission / insertion / weakForm / linking / flap / assimilation / reduction）を OSS Worker で完全に生成し、新たに epenthesis（母音挿入）と lexicalStress（語強勢誤り）を追加する |
| **受入基準** | connected speech 系 finding が一律 `connectedSpeech` ではなく現象別（weakForm / linking / flap / assimilation / reduction）に分類されること。母音挿入が substitution / insertion と区別された `epenthesis` として検出されること（E-11: 音節数の歪みとして独立検出）。語強勢の期待 vs 実測の不一致が `lexicalStress` として検出されること。enum 追加時に ACL スキーマ・API 型・ADR が同一 PR で更新されること |

#### REQ-103: 検出音素の NBest 診断 <a id="req-103"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | C-6, B-3, E-8 |
| **説明** | 「実際に何と発音したか」を確率的に診断する。期待音素ごとに実発話の音素候補（NBest）と信頼度を python-analyzer が返し、worker が confusion set（REQ-101）と突合して誤りパターンを同定する |
| **受入基準** | python-analyzer のレスポンスに期待音素ごとの実発話候補（上位 3 件以上 + 信頼度）が含まれること。worker の finding に「検出された音（最有力候補）」と「日本語話者の典型パターンに一致するか」の判定が含まれること（例: /æ/ → /a/ は典型射影、と同定できる）。候補が confusion set 外の場合も最有力候補をそのまま提示すること |

#### REQ-104: 明示的訂正フィードバック（3 層構造） <a id="req-104"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | E-5, C-2, B-2, REQ-015（基本要件） |
| **説明** | 各 finding のフィードバック文を「①何が起きたか（観測） ②なぜ起きるか（L1 干渉メカニズム） ③どう直すか（調音・練習指示）」の 3 層で生成する。生成はカタログ（REQ-101）駆動のテンプレートを基本とする |
| **受入基準** | ①が期待音 / 検出音 / 単語内位置を含むこと。②がカタログの L1 干渉説明を含むこと（例:「日本語のラ行は弾き音 [ɾ] のため、英語の /l/ と /r/ の両方がこの音に寄ります」）。③が調音点・調音法レベルの行動可能な指示を含むこと（例:「舌先を上前歯の裏の歯茎に当て、舌の両脇から息を流す」— B-2 の ELSA 水準）。phenomenon × 音素対立 × 単語内位置の組合せでメッセージが変化すること（固定文 4 種からの脱却）。日本語必須・英語併記は基本要件 REQ-NF-034/035 を継承 |

#### REQ-105: 調音指導コンテンツ <a id="req-105"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Should |
| **根拠** | 調査 §3.3-6, B-2 |
| **説明** | 主要音素（カタログの segmental 項目）ごとに、矢状断面図（舌・唇・軟口蓋の位置）と調音手順テキストを提供し、詳細パネルから参照できるようにする |
| **受入基準** | 高優先音素（/r/, /l/, /æ/, /ʌ/, /iː/-/ɪ/, /θ/, /ð/, /v/, /f/, schwa）の調音図解と手順が存在すること。詳細パネルの ③どう直すか（REQ-104）からリンクされること。視覚情報は音響フィードバック（録音再生・お手本再生）と併置されること（E: 視覚調音情報は音響と併用で初めて価値が出る — Chang 2023） |

#### REQ-106: LLM による個別化フィードバック文生成（オプション） <a id="req-106"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Could |
| **根拠** | E-5, 技術制約方針（OSS 基本 + 必要なら LLM 可） |
| **説明** | カタログ駆動テンプレート（REQ-104）を基本としつつ、LLM に解析結果（finding 一覧 + カタログ項目 + 履歴）を渡して文脈化されたフィードバック文・練習提案を生成するオプションを提供する |
| **受入基準** | LLM 無しでも REQ-104 が完全に機能すること（LLM は品質向上レイヤであり依存しない）。LLM 生成文がカタログのエビデンス整合性を逸脱しないようプロンプトにカタログ項目を埋め込むこと。外部 API 送信の明示（基本要件 REQ-NF-022 準拠） |

#### REQ-107: 細粒度データの API / UI 到達（配線修正） <a id="req-107"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must（最優先・即効） |
| **根拠** | C-3, C-5, REQ-014（基本要件）の未達解消 |
| **説明** | 計測済みだが UI に届いていないデータを配線する。workspace API が `phenomenon` / `gop` / `confidence` / サマリーメッセージを返し、詳細パネルが表示する |
| **受入基準** | workspace API のレスポンスに finding ごとの phenomenon / gop / confidence が含まれること。詳細パネルに信頼度・解析エンジン名・関連スコアへのリンクが表示されること（基本要件 REQ-014 の受入基準を完全達成）。エンジン別サマリーメッセージがワークスペース画面に表示されること。全音素の GOP 系列（閾値未満で finding 化されなかった音素を含む）を解析結果に保存し、単語単位のヒートマップ等で参照可能であること |

#### REQ-108: 信頼度表示と誤検出ヘッジ <a id="req-108"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | B-5（音素誤検出 precision ~60% / recall 40–80%） |
| **説明** | 自動検出の誤りリスクを UI で吸収する。低信頼度の指摘を断定調で提示しない |
| **受入基準** | finding の信頼度が 3 段階以上で可視化されること。低信頼度の指摘は表現をヘッジし（「〜の可能性があります」）、デフォルト表示から折りたたむこと。ユーザーが指摘を「誤検出として却下」でき、却下が履歴に残ること |

#### REQ-109: connected speech の現象別・位置特定指摘 <a id="req-109"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | C-4, REQ-102 |
| **説明** | 連結発話の指摘を「どの単語間で・どの現象が・どう発音されるべきか」まで具体化する |
| **受入基準** | 指摘が対象単語ペア（または機能語）を特定すること。現象別（linking / weakForm / flap / assimilation / reduction）のメッセージと期待発音（IPA、例: "want to" → /wɑnə/）が表示されること。固定文「ここは連結・弱形にするとネイティブらしくなります」が全廃されること |

### 4.2 カテゴリ B: 採点・評価

#### REQ-111: 二段階ゴールモデルに基づく採点 <a id="req-111"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | E-3, E-9, §2 |
| **説明** | Stage I（明瞭性）/ Stage II（ネイティブ性）の 2 軸スコアを提供する。明瞭性スコアは error gravity（FL 重み）付きで集計し、ネイティブ性スコアは現行の厳しめ判定を継承する |
| **受入基準** | 明瞭性スコアが FL 重み付けで計算されること（高 FL 誤りは加算的に減点、低 FL 誤りは件数が増えても減点が逓減 — Munro & Derwing 2006 の知見を反映）。CEFR 音韻統制の 3 下位尺度（全体 / 分節 / 韻律）に対応する内訳が表示されること。両スコアの定義差（なぜ明瞭性は高いがネイティブ性は低い、が起きるか）が UI で説明されること |

#### REQ-112: functional load に基づく優先度付けと focus sounds <a id="req-112"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | E-9, B-4, C-8 |
| **説明** | 検出された誤りを FL ランク × 出現頻度 × 習熟度で優先度付けし、「いま直すべき音」（focus sounds）リストを提示する |
| **受入基準** | 優先度が固定リストでなく三項（FL × 頻度 × 習熟度）から動的に計算されること。focus sounds が解析のたびに漸進更新されること（B-4: 再診断テストを別途課さない）。低 FL 誤り（/θ/ 等）は「検出するが優先度ラベルは低、理由を FL で説明する」表示であること |

#### REQ-113: 習熟度適応の優先度切り替え <a id="req-113"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Should |
| **根拠** | E-10 |
| **説明** | 初中級ユーザーには韻律 + 母音挿入 + 高 FL 分節を、上級ユーザーには母音の質の精緻化を前面に出す |
| **受入基準** | 習熟度推定（診断結果 + スコア履歴）に応じて focus sounds の構成比が変わること。切り替えロジックが設計書に明文化されること |

#### REQ-114: 韻律計測の拡充 <a id="req-114"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | E-10（韻律は全習熟度帯で comprehensibility に効く）, T-3, C-4 |
| **説明** | 現行の代理指標（無音 + schwa 実現 + 話速）を、F0 輪郭・語強勢・リズム・弱形の直接計測に拡充する |
| **受入基準** | F0 輪郭（学習者 + 参照）が解析結果に含まれること。語強勢の期待値（辞書由来 0/1/2）と実測値（F0 / 強度 / 持続時間からの推定）の突合せが単語単位で返ること（B-3 の Speechace `stress_level` / `predicted_stress_level` 相当）。リズム指標（nPVI 等）が算出され、モーラ拍リズムへの偏りが指摘されること。機能語の弱形実現率が単語特定付きで返ること |

#### REQ-115: 母音挿入（epenthesis）の独立検出 <a id="req-115"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | E-11（時間構造補正だけで明瞭性 +19pt）, E-8 |
| **説明** | 子音クラスター内・語末子音後の母音挿入を、音素置換とは別カテゴリ（音節数・時間構造の逸脱）として検出する |
| **受入基準** | 期待音節数と実測音節数の差分が単語単位で検出されること。挿入位置と挿入母音（[ɯ] / [o] / [i]）が特定されること。フィードバック文が「カタカナ読みの混入」という L1 メカニズムと「音声を先に聞いてから模倣する」対策（E-11: 模倣時は挿入が減る）を含むこと |

#### REQ-116: 採点エンジンの進化余地の確保 <a id="req-116"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Should |
| **根拠** | E-6, T-1, 基本要件 REQ-NF-050 |
| **説明** | 古典 GOP（現行）から GOPT / SSL ベース採点（音素 PCC 0.45 → 0.61+）へ置換可能なよう、採点器を contract で抽象化する |
| **受入基準** | 採点器の入出力契約（音声 + 期待音素列 → 音素/単語/発話スコア + NBest）が ACL 層で固定され、実装（Kaldi GOP / GOPT / SSL）を差し替えても上位層が変更不要であること。採点器名・バージョンが解析結果に保存されること（基本要件 REQ-008 継承） |

### 4.3 カテゴリ C: 訓練ループ

#### REQ-121: 診断テストと弱点プロファイル <a id="req-121"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | B-4, B-2 |
| **説明** | 短時間（2〜5 分）の診断テスト（カタログの主要対立を網羅する読み上げ文セット）で弱点プロファイルを初期化し、以後は日常の解析結果で漸進更新する |
| **受入基準** | 診断用文セットがカタログの高 FL 対立 + 母音挿入 + 韻律を網羅すること。診断結果から focus sounds（REQ-112）と推奨訓練（REQ-101 の訓練種別マッピング）が生成されること。プロファイルがセクション練習の解析結果でも更新されること |

#### REQ-122: HVPT 知覚訓練モジュール <a id="req-122"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | 調査 §3.3-1,2（本調査で最もエビデンスが厚い手法）, T-7 |
| **説明** | focus sounds の音素対立に対する高変動知覚訓練（多話者ミニマルペア識別課題）を提供する |
| **受入基準** | 識別課題（多択 forced-choice）形式であること（弁別課題でないこと: g 0.95 vs 0.57）。刺激が話者 5 名以上・男女混在・複数音韻文脈（語頭 / 語中 / クラスター）であること（単一話者では般化しない）。試行ごとに正誤フィードバック + 正解音再生があること（産出転移 g 0.94 vs 0.45）。応答ラベルが綴り / キーワード / IPA であること（画像不可）。1 セッション 20–30 分で区切られ、累計訓練時間が記録されること（300–400 分で効果が頭打ちになる旨を進捗に反映）。刺激音源は VCTK / LibriTTS（CC BY 4.0）等の自然音声を優先すること |

#### REQ-123: ミニマルペア産出ドリル <a id="req-123"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | 調査 §3.3-2（知覚訓練単独の産出転移は g≈0.49 に留まる → 産出練習の併設が必要） |
| **説明** | 知覚訓練（REQ-122）と対になる産出ドリル。ミニマルペア・短文を録音し、対象音素に絞った即時評価を返す |
| **受入基準** | focus sounds に対応するミニマルペア・例文がカタログから生成されること。録音への評価が対象音素の GOP / NBest 診断に絞って即時（数秒以内）に返ること。知覚 → 産出の順でセッションが構成されること |

#### REQ-124: お手本音声の生成と再生 <a id="req-124"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must（基本要件の対象外指定を撤回） |
| **根拠** | §3, T-5, B-2 |
| **説明** | セクション本文・指摘箇所・ドリル例文のお手本音声を OSS TTS で生成し、全文 / 指摘箇所部分 / 速度調整付きで再生できる |
| **受入基準** | General American の TTS（第一候補 Kokoro-82M, Apache-2.0）でお手本を生成できること。指摘箇所（finding の text range）単位の部分お手本再生ができること。再生速度を 0.5–1.0x で調整できること。学習者録音とお手本を交互再生（A/B 比較）できること。CPU で実用速度（セクション生成が録音解析と同程度以内）であること |

#### REQ-125: シャドーイングモード <a id="req-125"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Should |
| **根拠** | 調査 §3.3-3（韻律・流暢性・comprehensibility に d≈0.8–0.9、日本人 EFL で実証） |
| **説明** | お手本音声（REQ-124）に重ねて発話するシャドーイング練習を提供し、韻律・流暢性の訓練に充てる |
| **受入基準** | お手本再生と同時録音ができること。ラグ（お手本との時間ずれ）が計測・表示されること（ラグ過大の学習者には効果がないため、スロー再生から開始する導線があること）。評価フォーカスが韻律・流暢性（リズム / ポーズ / 話速）であること（分節の細評価はしない）。週次の実施回数が進捗に記録されること（週 3–4 回 × 10–15 分の推奨をガイドすること） |

#### REQ-126: ピッチ・韻律可視化 <a id="req-126"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Should |
| **根拠** | 調査 §3.3-5, REQ-114 |
| **説明** | 学習者と参照音声の F0 輪郭を同一グラフに重ねて表示し、韻律の気づきを与える |
| **受入基準** | 連続 F0 輪郭の重ね描きであること（記号表記主体でないこと: タイミングエラー 3–10% vs 40–50%）。発話直後の事後表示であること（発話中のリアルタイム表示は必須としない）。語強勢の期待 / 実測（REQ-114）が輪郭上にマークされること。表示を隠して自己評価してから答え合わせする「ブラインドモード」があること |

#### REQ-127: 分散学習スケジューラ <a id="req-127"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Should |
| **根拠** | 調査 §3.3-4（スペーシングは集中練習の約 2 倍の遅延効果） |
| **説明** | 訓練（REQ-122/123）と復習を等間隔スペーシングでスケジュールする |
| **受入基準** | 同一対立の訓練セッションが 24 時間程度の等間隔で提案されること（拡張間隔は必須としない: 等間隔と拡張に有意差なし）。セッション内正答率が 60% に達するまで間隔を空けないこと（最低習得強度ゲート）。1 セッションが 20–30 分で打ち切られること |

#### REQ-128: Golden speaker（自分の声のネイティブ風お手本） <a id="req-128"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Could（検証フェーズ付きオプション） |
| **根拠** | 調査 §3.3-7（RCT なし・エビデンス弱、ただし差別化候補）, T-6 |
| **説明** | 学習者自身の声質でネイティブ発音を合成したお手本（RVC / kNN-VC）を提供する |
| **受入基準** | GPU 任意環境でのみ有効化されるオプション機能であること。セグメント変換を含むこと（韻律のみ変換は効果なし — Felps 2009）。「自己 GS / ネイティブ TTS / 自録音」を切替比較でき、利用ログを残して効果検証に使えること。合成品質が実用水準（明瞭に聞き取れる）に達しない場合は提供しないゲートを設けること |

#### REQ-129: 進捗可視化と学習継続設計 <a id="req-129"></a>

| 項目 | 内容 |
|---|---|
| **優先度** | Must |
| **根拠** | B-4, E-12, E-2 |
| **説明** | 弱点の解消過程を可視化し、学習継続を支える |
| **受入基準** | focus sounds ごとのスコア推移（時系列）が表示されること。CEFR 3 下位尺度（全体 / 分節 / 韻律）に対応する 2 軸以上のレーダー・推移表示があること。訓練の累計時間・実施間隔が表示されること。同一セクションの過去録音との比較再生ができること（基本要件 REQ-018 を拡張）。効果測定は統制課題（同一文の再録音・ドリル正答率）の素性別スコアで行うこと（E-2: 自発発話への転移を過大評価しない） |

## 5. 非機能要件

| ID | 要件 | 詳細 |
|---|---|---|
| REQ-NF-101 | OSS ライセンス制約 | 本番組込みは Apache-2.0 / MIT / BSD / CC BY を基本とする。GPL 系（phonemizer, seed-vc, Piper 後継, parselmouth※）はサービス境界で隔離するか代替する（※parselmouth は python-analyzer 内利用に留め、配布形態を踏まえてライセンス影響を ADR で判断する）。L2-ARCTIC（CC BY-NC）は本番再配布しない。XTTS-v2（CPML）は採用しない |
| REQ-NF-102 | CPU 基本 / GPU 任意 | 採点・お手本生成・訓練は CPU で成立すること（Kaldi GOP + GOPT / MFA / Kokoro はいずれも CPU 可）。Golden speaker（REQ-128）等の GPU 前提機能はオプションとして分離する |
| REQ-NF-103 | フィードバック文の研究整合性 | カタログ項目・フィードバック文言はエビデンス ID（調査 C/E/T/B）にトレース可能であること。調査 §3.5 の棄却済み主張を根拠にしないこと |
| REQ-NF-104 | 日本語 L1 データ制約の明示 | 公開ベンチマーク（speechocean762 / L2-ARCTIC）に日本語話者が含まれない制約（T-4）を認識し、採点閾値は自己録音キャリブレーション + confusion set ルール（E-8）で補正する。将来の自前データ収集を見据え、ユーザーの却下操作（REQ-108）と録音を学習データ候補として保存できる構造にする |
| REQ-NF-105 | 誤フィードバック防止 | 音素レベル自動検出の precision 限界（~60%、B-5）を前提に、断定的な誤指摘でユーザーの正しい発音を壊さない設計とする（信頼度ゲート + ヘッジ表現 + 却下導線） |
